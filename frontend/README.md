# ShieldFlow — Confidential Escrow on Zama fhEVM

ShieldFlow is a **privacy-preserving milestone escrow** built on Ethereum Sepolia using [Zama's fhEVM](https://docs.zama.ai/fhevm). Deposit amounts, released balances, and per-milestone allocations are stored as `euint64` FHE ciphertexts on-chain — only the authorized client, contractor, or auditor can decrypt them.

Built for **Zama Season 3** hackathon.

---

## Key FHE Usage

### 1. Encrypted Storage

Every monetary value in the contract is stored as an `euint64` FHE ciphertext, never as a plaintext balance:

```solidity
struct Escrow {
    euint64 totalDeposit;   // total funded amount — FHE-encrypted
    euint64 releasedAmount; // running released total — FHE-encrypted
    ...
}
struct Milestone {
    euint64 encryptedAmount; // per-milestone allocation — FHE-encrypted
    ...
}
```

### 2. FHE Encryption on Deposit

When the client calls `deposit()`, they submit `externalEuint64` handles (generated by the relayer SDK in the browser) plus a zero-knowledge input proof. The contract calls `FHE.fromExternal()` to convert them to on-chain ciphertexts:

```solidity
euint64 encTotal = FHE.fromExternal(encryptedTotal, inputProof);
e.totalDeposit = encTotal;
FHE.allowThis(e.totalDeposit);        // contract can operate on it
FHE.allow(e.totalDeposit, client);    // client can decrypt
FHE.allow(e.totalDeposit, contractor); // contractor can decrypt
```

The ACL (`FHE.allow`) controls who can later decrypt each handle.

### 3. FHE Arithmetic on Release

When the client releases a milestone, the contract performs **encrypted addition** on the released total — the running sum stays encrypted throughout:

```solidity
euint64 newReleased = FHE.add(e.releasedAmount, m.encryptedAmount);
// Overflow guard — select old value if addition wrapped
ebool didOverflow = FHE.lt(newReleased, e.releasedAmount);
e.releasedAmount = FHE.select(didOverflow, e.releasedAmount, newReleased);
// Re-grant ACL so the updated handle is readable by authorized parties
FHE.allowThis(e.releasedAmount);
FHE.allow(e.releasedAmount, client);
FHE.allow(e.releasedAmount, contractor);
```

No intermediate plaintext is ever produced on-chain.

### 4. Auditor ACL Grant

A client can grant a third-party auditor read access to all encrypted handles via `grantAuditorAccess()`. This calls `FHE.allow()` for each existing ciphertext, giving the auditor the ability to request decryption through the KMS without touching ETH transfers:

```solidity
FHE.allow(e.totalDeposit, auditor);
FHE.allow(e.releasedAmount, auditor);
for (uint8 i = 0; i < e.milestoneCount; i++) {
    FHE.allow(e.milestones[i].encryptedAmount, auditor);
}
```

### 5. Client-Side Decryption (userDecrypt)

The frontend uses the Zama relayer SDK's `userDecrypt` flow so only authorized users can view amounts:

1. Generate an ephemeral ML-KEM keypair in the browser
2. Build an EIP-712 request binding the public key to the contract and a 1-day time window
3. User signs the EIP-712 message with MetaMask — proves wallet identity to the KMS
4. Send `(handle, contractAddress, publicKey, signature)` to the Zama relayer
5. KMS checks the on-chain ACL, re-encrypts the plaintext with the user's ephemeral public key, returns the ciphertext
6. Browser decrypts locally with the ephemeral private key → plaintext `bigint`

See `src/lib/fhe/fhevm.ts` — `decryptHandle()`.

## User Flow

### Client
1. **Create** — Enter contractor address (optional auditor), add 1–10 milestones with ETH amounts and deadlines
2. **Fund** — Sign two wallet transactions: (a) register escrow, (b) deposit ETH with FHE-encrypted amounts
3. **Manage** — Approve contractor submissions, release funds per milestone, grant auditor access
4. **Reveal** — Click the 🔒 lock icon on any amount → sign EIP-712 → see decrypted ETH value

### Contractor
1. Visit the escrow link shared by client
2. **Submit** milestone when work is complete
3. **Withdraw** ETH once client releases the milestone
4. **Reveal** their own milestone amounts via FHE decrypt

### Auditor
1. Client grants access via `grantAuditorAccess` (on-chain ACL update)
2. Visit the escrow page — banner shows "Auditor view — FHE decryption access"
3. **Reveal** any encrypted amount — the relayer verifies the ACL before decrypting

## FHE Privacy Model

| Data | On-chain storage | Who can decrypt |
|---|---|---|
| Total deposit | `euint64` ciphertext | Client, Contractor, Auditor (if granted) |
| Released amount | `euint64` ciphertext | Client, Contractor, Auditor |
| Per-milestone amount | `euint64` ciphertext | Client, Contractor, Auditor |
| Parties (client/contractor/auditor) | Plaintext `address` | Everyone |
| ETH transfer amounts | Plaintext `uint256` (for Solidity transfers) | Everyone |

> **Note:** `plainAmountWei` is stored in plaintext because Solidity must move real ETH. The `euint64` ciphertexts carry the same values confidentially for the FHE layer — demonstrating the encrypt/decrypt cycle.
